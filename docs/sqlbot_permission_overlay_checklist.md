# SQLBot 权限能力叠加改造清单

## 目标

- 在 `awesome-project` 中落地“认证 + 授权 + 数据权限”三层能力。
- 与 SQLBot 保持核心模型一致（`oid`、角色、资源归属、行列权限），降低后续迁移成本。
- 采用分阶段上线，优先消除硬编码用户与越权风险。

## 范围

- 后端：`src/system`、`src/common`、`src/chat`、`src/datasource`
- 前端：`frontend-react/pages/construct`、`frontend-react/src/components/layout`
- 文档：本文件作为执行与验收清单

## 阶段一：认证与会话统一（P0）

- [ ] 新增统一 `CurrentUser` 依赖（替代 `current_user_id=1`）。
- [ ] 新增 JWT 解析中间件或全局依赖方案（二选一，统一到一个入口）。
- [ ] `chat` 全接口接入真实登录用户上下文。
- [ ] `datasource` 全接口接入真实登录用户上下文。
- [ ] 前端请求统一注入 `Authorization: Bearer <token>`。
- [ ] 补充认证失败统一错误码与错误文案（401/403）。

## 阶段二：角色与资源权限（P0/P1）

- [ ] 定义角色模型：`admin`、`ws_admin`、`member`。
- [ ] 新增角色校验装饰器/依赖：`require_permission(...)`。
- [ ] 新增资源归属校验：数据源、对话、记录按 `oid` 或 `owner` 限制。
- [ ] 所有关键写操作接口（新增/修改/删除）加角色校验。
- [ ] 所有关键读操作接口（详情/列表）加资源过滤。
- [ ] 增加至少 1 组越权测试用例（他人资源读取/修改应失败）。

## 阶段三：数据权限（行/列）（P1）

- [ ] 设计权限规则表：列禁用、行过滤规则树。
- [ ] 生成器执行前注入列过滤逻辑（不可见字段剔除）。
- [ ] SQL 执行前注入行级过滤条件（按用户变量/系统变量）。
- [ ] 对高风险 SQL 场景增加白名单与审计日志。
- [ ] 增加规则命中日志，支持问题追踪与审计导出。

## 阶段四：管理界面（P1）

- [ ] 新增“权限管理”导航入口。
- [ ] 新增角色管理页（用户-角色映射）。
- [ ] 新增资源授权页（用户/角色 -> 数据源、对话范围）。
- [ ] 新增数据权限页（行级规则、列级可见性）。
- [ ] 新增权限预览能力（给定用户查看最终有效权限）。

## 接口改造建议（后端）

- [ ] `POST /system/login`：返回 token 与最小用户信息。
- [ ] `GET /system/me`：返回用户、角色、oid、权限摘要。
- [ ] `GET/POST /permission/roles`：角色管理。
- [ ] `GET/POST /permission/grants`：资源授权管理。
- [ ] `GET/POST /permission/data-rules`：行列权限规则管理。
- [ ] `POST /permission/effective`：权限生效预览。

## 前端改造建议

- [ ] 新增 `pages/construct/permission` 路由组。
- [ ] 新增权限相关 API 模块（`src/api/permission.ts`）。
- [ ] 菜单接入“权限管理”并与现有构建页风格一致。
- [ ] 表格页统一支持：搜索、筛选、分页、编辑弹窗。
- [ ] 增加“未接入后端时”的空态与提示文案，便于并行开发。

## 验收标准

- [ ] 未登录无法访问受保护接口（401）。
- [ ] 普通用户无法访问管理员接口（403）。
- [ ] 用户无法读取/修改非授权数据源与对话。
- [ ] 行列权限生效后，SQL 结果字段与行范围符合预期。
- [ ] 前端权限管理页可完成增删改查的联调流程。

## 风险与注意事项

- 行级规则 SQL 拼接需防注入，优先 AST 方式改写。
- 避免在接口层散落权限逻辑，统一依赖/装饰器治理。
- 从“最小权限”默认策略启动，避免放开式授权。
- 保留权限变更审计日志，便于回溯问题。

## 执行顺序建议

1. 先完成阶段一与阶段二（基础安全闭环）。
2. 同步推进阶段四前端页面骨架，提前可视化管理入口。
3. 最后上线阶段三数据权限，并逐步灰度。
