# SQLBot 权限能力叠加改造清单

## 目标

- 在 `awesome-project` 中落地“认证 + 授权 + 数据权限”三层能力。
- 与 SQLBot 保持核心模型一致（`oid`、角色、资源归属、行列权限），降低后续迁移成本。
- 采用分阶段上线，优先消除硬编码用户与越权风险。

## 范围

- 后端：`src/system`、`src/common`、`src/chat`、`src/datasource`
- 前端：`frontend-react/pages/construct`、`frontend-react/src/components/layout`
- 文档：本文件作为执行与验收清单

## 阶段一：认证与会话统一（P0）

- [x] 新增统一 `CurrentUser` 依赖（替代 `current_user_id=1`）。
- [x] 统一 JWT 登录链路（`/system/login` + `/system/me` + 全局路由守卫）。
- [ ] `chat` 全接口接入真实登录用户上下文（仍有历史逻辑待清理）。
- [ ] `datasource` 全接口接入真实登录用户上下文（当前为接口可用，细粒度权限待收口）。
- [x] 前端请求统一注入 `Authorization: Bearer <token>`。
- [x] 认证失败统一清 token 并跳转登录页（前端行为已完成）。

- [x] 新增资源归属查询接口（工作空间成员/资源映射）。
- [ ] 所有关键写操作接口加角色校验（当前为可用版本，严格校验仍需补齐）。
- [ ] 所有关键读操作接口加资源过滤（部分完成）。
- [ ] 增加至少 1 组越权测试用例（他人资源读取/修改应失败）。

## 阶段三：数据权限（行/列）（P1）

- [x] 落地规则表：`ds_rules`、`ds_permission`。
- [x] 支持规则组 + 规则项（行/列）保存与加载（`/ds_permission/list|save|delete`）。
- [x] 前端支持列级字段开关、行级条件编辑（含多条件 `AND/OR`）。
- [ ] SQL 执行前注入行级过滤条件（按用户变量/系统变量）。
- [ ] 对高风险 SQL 场景增加白名单与审计日志。
- [ ] 增加规则命中日志，支持问题追踪与审计导出。

## 接口改造建议（后端）

- [ ] `POST /system/login`：返回 token 与最小用户信息。
- [ ] `GET /system/me`：返回用户、角色、oid、权限摘要。
- [ ] `GET/POST /permission/roles`：角色管理。
- [ ] `GET/POST /permission/grants`：资源授权管理。
- [ ] `GET/POST /permission/data-rules`：行列权限规则管理。
- [ ] `POST /permission/effective`：权限生效预览。

## 验收标准（当前状态）

- [x] 未登录访问受保护页面会自动跳转登录页。
- [ ] 普通用户无法访问管理员接口（需补后端严格角色校验）。
- [ ] 用户无法读取/修改非授权数据源与对话（需补资源过滤闭环）。
- [ ] 行列权限生效到查询执行链路（当前仅完成规则配置与持久化）。
- [x] 前端权限管理页可完成主要增删改查联调流程。

## 手工验收清单（可直接执行）

1. 登录：访问 `/login`，用管理员账号登录，确认可回跳到目标页。
2. 工作空间：在“工作空间管理”中新增空间、重命名、切换空间。
3. 成员：给空间添加成员，切换成员类型（普通/管理员），移除成员。
4. 角色与资源：打开“角色管理”“资源授权”，确认均为真实接口数据（无 fallback）。
5. 数据权限：
   - 新建规则组；
   - 步骤一添加多个规则项（行/列）；
   - 行权限配置多个条件并切换 `AND/OR`；
   - 列权限按字段开关；
   - 步骤二选择受限用户并保存；
   - 刷新页面后确认回显一致。

## 风险与注意事项

- 行级规则 SQL 拼接需防注入，优先 AST 方式改写。
- 避免在接口层散落权限逻辑，统一依赖/装饰器治理。
- 从“最小权限”默认策略启动，避免放开式授权。
- 保留权限变更审计日志，便于回溯问题。

## 执行顺序建议

1. 先完成阶段一与阶段二（基础安全闭环）。
2. 同步推进阶段四前端页面骨架，提前可视化管理入口。
3. 最后上线阶段三数据权限，并逐步灰度。
